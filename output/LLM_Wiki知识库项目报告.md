## 一、什么是 LLM Wiki

### 1.1 定义

**LLM Wiki** 是一种基于大语言模型（LLM）驱动的**增量式知识库编译系统**，核心理念源自 Andrej Karpathy 的"Software 2.0"思想——将知识整理过程从人工编写转变为 AI 自动化编译。

### 1.2 核心概念

| 概念 | 说明 |
|------|------|
| **双向链接** | 使用 `[[概念]]` 语法建立概念间关联，形成知识网络 |
| **原子化** | 每个知识点独立成文，便于理解和复用 |
| **可追溯** | 所有知识标注来源，拒绝幻觉 |
| **图形化渲染** | 通过 Obsidian 等工具可视化知识图谱 |
| **增量编译** | 持续迭代而非一次性构建 |

### 1.3 与传统知识库的区别

| 维度 | 传统知识库 | LLM Wiki |
|------|------------|----------|
| **构建方式** | 人工编写、维护 | AI 自动编译、持续增量 |
| **知识组织** | 分类目录 | 双向链接网络 |
| **检索方式** | 关键词搜索 | 概念关联 + 图谱浏览 |
| **维护成本** | 高（需专人维护） | 低（AI 自动维护） |
| **知识溯源** | 难追踪 | 脚注标注来源文件 |
| **可视化** | 平面文档 | 交互式知识图谱 |

### 1.4 设计哲学

采用 Karpathy 模式的核心原则：

1. **自动化优先**：AI 负责编译、链接、维护，人类负责审查和微调
2. **增量迭代**：新素材持续融入，不破坏已有结构
3. **知识网络**：概念间建立双向链接，而非孤立文档
4. **可验证性**：每条知识可追溯到原始素材，确保准确性

## 二、知识库构建目标

### 2.1 核心目标

将公司积累的技术笔记、文档、论文等原始素材，编译为**结构化、可检索、可交叉引用**的知识网络，服务于：

| 目标 | 说明 |
|------|------|
| **知识沉淀** | 将散落的笔记转化为系统化知识体系 |
| **快速检索** | 支持概念级精准查找而非关键词模糊搜索 |
| **知识传承** | 新员工可通过图谱快速理解技术栈全貌 |
| **协作共享** | 团队共享统一知识库，避免信息孤岛 |
| **持续演进** | 新素材自动融入，知识库随业务增长 |

### 2.2 目标用户

| 用户群体 | 使用场景 |
|----------|----------|
| **研发工程师** | 查询技术概念、理解系统架构、学习新技术 |
| **新员工** | 通过知识图谱快速了解技术栈 |
| **架构师** | 查阅跨领域概念关联、做技术决策 |
| **技术 Leader** | 评估知识覆盖度、发现知识缺口 |

### 2.3 预期价值

| 价值维度 | 量化目标 |
|----------|----------|
| **检索效率** | 从 30 分钟（查找文档）到 1 分钟（概念直达） |
| **知识复用** | 减少重复解答同一问题 70% |
| **学习曲线** | 新员工熟悉技术栈时间缩短 50% |
| **知识覆盖** | 核心技术领域覆盖率 > 90% |

## 三、SKILL 规约

### 3.1 规约定义

`SKILL.md` 是知识库架构师的**最高行为准则**，定义了 AI 在此工作空间的行为规范，确保：

- 自动化、增量化的维护方式
- 符合 Obsidian 渲染要求的知识结构
- 可追溯、可验证的知识质量

### 3.2 物理架构

工作空间严格划分为**两层物理结构**：

| 目录 | 角色 | Agent 权限 | 内容类型 |
|------|------|------------|----------|
| `raw/` | 输入层（原始素材库） | **绝对只读** | 论文、网页剪藏、随笔、播客逐字稿 |
| `wiki/` | 输出层（结构化知识库） | **可读、可写、可精炼** | 原子化、带双向链接的 Markdown 文件 |

**权限分离原则**：raw/ 只读确保原始素材不被破坏，wiki/ 可写允许持续优化。

### 3.3 核心任务

#### 任务一：增量编译

扫描 `raw/` 目录中**新存入或修改**的素材，执行：

| 子任务 | 说明 |
|--------|------|
| **概念提取** | 提取核心实体、技术名词、关键论点 |
| **智能合并** | 存在同名文件则增量合并，不存在则创建新文件 |
| **链接建立** | 使用 `[[概念]]` 语法建立双向链接 |
| **来源标注** | 脚注标注原始素材路径 |

**禁止**：无状态的全量重写（破坏人类微调内容）

#### 任务二：静态检查

执行知识库健康检查，确保：

| 检查项 | 说明 |
|--------|------|
| **消灭孤立节点** | 为悬空链接创建概念骨架 |
| **建立强关联** | 为逻辑相关的孤立节点建立链接 |
| **链接有效性** | 检查链接目标文件是否存在 |

### 3.4 生成规范

所有 Wiki 文件必须遵守**四条高标准线**：

#### 规范一：严格原子化

一个 Markdown 文件**只讲透一个核心概念**。

```
正例: wiki/SVA.md 只讲 Shared Virtual Addressing
反例: wiki/虚拟化技术.md 包含 KVM、VFIO、热迁移等多个概念
```

**原因**：原子化确保 Obsidian 关系图谱聚焦清晰。

#### 规范二：标准双向链接语法

提及任何核心概念必须使用 `[[概念]]` 语法：

```
正例: 在 ARM 架构下，配置依赖于 [[ACPI]] 协议，而非 [[Device Tree]]。
反例: 在 ARM 架构下，配置依赖于 ACPI 协议。
```

**效果**：Obsidian 自动渲染概念间的关联线条。

#### 规范三：严苛的可追踪性

每段核心事实末尾追加脚注 `[^1]`，文末标注来源：

```markdown
基于 MSI-X 的动态分配机制能够显著降低虚拟化环境下的中断延迟 [^1]。

[^1]: 来源于原始素材 raw/pci_passthrough_paper.pdf 第 4 页。
```

**目的**：拒绝幻觉，确保知识可验证。

#### 规范四：规范的 Frontmatter 结构

所有文件头部包含 YAML 元数据：

```yaml
---
type: knowledge-node
tags: [core-concept, auto-generated, iommu, virtualization]
last_compiled: 2026-05-22
---
```

**用途**：Obsidian 检索和分类。

### 3.5 Skill 配置

已将 SKILL.md 内容封装为 OpenCode Skill：

```
.opencode/skills/wiki-architect/SKILL.md
```

新 session 自动加载，具备：

- 知识库查询能力
- 增量编译能力
- 静态检查能力

### 5.3 知识网络示例

```

  无锁队列 → CAS → ABA问题 → Memory Order → 内存屏障 → ARM内存屏障

  SVA → SMMU → IOMMU → VFIO → KVM → vSVA → 热迁移 → Postcopy迁移

  伙伴系统 → 内存迁移 → 透明大页 → 内存规整 → LRU链表 → 内存回收

调度链路:
  CFS调度器 → PELT → 负载均衡 → 调度域 → 调度组 → 组调度

CPU微架构链路:
  超标量处理器 → ROB → 寄存器重命名 → 分支预测 → Cache → TLB

PCI链路:
  PCIe热插拔 → SR-IOV → VFIO → AER → DPC → FLR → ACS
```

| 问题 | 说明 |
|------|------|
| **缺少索引机制** | 无 `index.json` 快速检索入口 |
| **主题交叉不足** | 跨主题关联较少（虚拟化 ↔ 内存管理 ↔ IOMMU） |

| 限制 | 说明 |
|------|------|

| **版本依赖** | 知识基于特定版本（如 Linux v6.0），可能过时 |
| **语言限制** | 中文优先，英文素材处理效果可能不同 |
| **上下文限制** | 单次处理文件数量受上下文长度限制 |
