# Walk恢复

寄存器重命名资源恢复方法，根据ROB保存的信息反向一步步回退到分支点状态。

## 原理

ROB保存每个指令的"前一个映射的物理寄存器"：
- 指令D: `a5->p6,p1` (a5当前映射p6，之前映射p1)
- 指令C: `a0->p4,p0` (a0当前映射p4，之前映射p0)

Walk恢复：
- 根据ROB信息反向修改map表
- 把当前映射改回前一个映射
- 一步步回退到分支指令时的map

示意图：
```
      +-------------------------------------------------------------------------+
ROB  |      D       |      C       |      B       |      jump    |             |
      |  a5->p6,p1   |   a0->p4,p3  |   a2->p2,p5  |   a0->p0,p7  |             |
      +-------------------------------------------------------------------------+

当前map3: a5->p6, a0->p4, a2->p2
恢复步骤:
  1. D: a5改映射p1 -> map: a5->p1, a0->p4, a2->p2
  2. C: a0改映射p3 -> map: a5->p1, a0->p3, a2->p2
  3. B: a2改映射p5 -> map: a5->p1, a0->p3, a2->p5
得到jump rename后的map
```

## 优点

- 无需额外存储：利用ROB已有信息
- 硬件开销小：无需checkpoint硬件
- 适合浅流水线：恢复延迟可接受

## 缺点

- 恢复速度慢：需遍历多条指令
- 每条指令都需修改map
- 深流水线恢复延迟大

## 与Checkpoint恢复对比

Checkpoint恢复：
- 一次拷贝完成
- 需额外checkpoint存储
- 恢复速度快

Walk恢复：
- 一步步反向回退
- 无需额外存储
- 恢复速度慢

## ROB信息

ROB需保存：
- 输出架构寄存器的当前物理寄存器映射
- 输出架构寄存器的前一个物理寄存器映射
- rename时写入ROB
