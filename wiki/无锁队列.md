# 无锁队列

无锁队列是基于原子操作实现的并发队列，避免了传统锁机制带来的性能开销[^1]。

## 实现方式

队列可以基于链表或数组实现[^1]：
- **链表实现**：最大问题是引入 ABA 问题
- **数组实现**：也有 ABA 问题，但更常见；需要一开始分配全部队列内存

## 数组实现核心逻辑

基于数组实现的核心逻辑[^1]：

```
+---+---+---+---+---+---+---+---+
|   | x | x | x | x |   |   |   |
+---+---+---+---+---+---+---+---+
      ^               ^ 
     read          commit
                      ^
                    write
```

三个关键指针：
- **write**：push 写入数据的位置
- **read**：pop 读取数据的位置  
- **commit**：彻底完成 copy 数据的位置

Push 流程[^1]：
1. 原子抢队列位置（用 [[CAS]] 操作）
2. 把数据 copy 到占有的位置
3. 更新 commit（需等待前面位置都 commit 完成）

Pop 流程[^1]：
1. 检查 read 和 commit 是否相等（队列空判断）
2. 原子更新 read（用 [[CAS]] 操作）
3. 返回数据

## 关键问题

### CAS 编码错误

在 commit_pos 的 CAS 操作中，如果 CAS 失败，`curr_write` 值会被更新为 `commit_pos` 的值，导致逻辑混乱[^1]。

**解决办法**：每次做 CAS 之前把 `curr_write` 的值先保存，如果 CAS 失败，用保存的值恢复[^1]。

### ABA 问题

**write_pos ABA 问题**[^1]：

Thread1 push 在执行 write_pos 的 CAS 前 sleep，期间其他线程持续做 pop/push。当 Thread1 醒来时，write_pos 和 curr_write 可能相等（刚好回绕一圈），CAS 成功，但队列实际已满。

**read_pos ABA 问题**[^1]：

Thread1 pop 执行完读数据后 sleep，期间其他线程不断 push/pop 数据，队列数据更新一轮。Thread1 醒来时 read_pos 相同，CAS 成功，但读到了错误/重复的数据。

**解决办法**：给指针加上版本号，每一轮版本号增加 1。CAS 操作时也要判断版本号[^1]。

### 未定义行为问题

两个 C 语言未定义行为问题[^1]：

1. **push/pop 读写队列数据**：从 C 语言角度看不一定是原子行为
   - 解决：使用 atomic_load/atomic_store（可用 `__ATOMIC_RELAXED`）

2. **队列版本号溢出**：使用有符号数溢出是未定义的
   - 解决：改成无符号数，且队列长度需是 2 的幂次

## 队列容量

在当前判断逻辑下，队列最大容量是 `QSIZE - 1`[^1]：
- read 和 commit 相等 → 队列为空
- write 的下一个位置是 read → 队列为满
