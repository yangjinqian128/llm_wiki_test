# 指令发射

将重命名后的指令分发到执行单元的issue queue，等待操作数ready后发射执行。

## 基本流程

```
rename → dispatch → issue queue → issue → EX
```

## Issue Queue类型

| 维度 | 类型 | 特点 |
|------|------|------|
| 组织 | 集中式 | 资源利用率高，实现复杂 |
| 组织 | 分布式 | 实现简单，资源可能浪费 |
| 数据 | 捕获式 | 进入时保存物理寄存器值 |
| 数据 | 非捕获式 | 发射时读取物理寄存器 |
| 移动 | 压缩式 | 发射后指令移动，功耗高 |
| 移动 | 非压缩式 | 固定位置，功耗低 |

## 仲裁逻辑

选择年龄最老的就绪指令发射，使依赖链前端先执行。

## 唤醒与优化

- 写回阶段或旁路网络唤醒依赖指令
- 尽早唤醒，让无依赖指令先运行
- 对拍数不确定指令（如load）可投机唤醒

## Replay

输入数据错误时可replay而非flush：
- 留在issue queue重新发射
- 省去fetch/decode/rename步骤
- 但可能挤占issue queue位置
